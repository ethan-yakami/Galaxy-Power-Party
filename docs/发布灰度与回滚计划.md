# 发布灰度与回滚计划（架构优先三阶段）

## 目标

在不引入 breaking change 的前提下，分阶段上线连接链路与 battle 启动优化，确保每一阶段都满足：

- 可观测（时序与错误分类可见）
- 可回滚（有开关与稳定回退版本）
- 可验证（上线前后有明确门禁）

## 阶段 1：架构收口（低风险重构）

### 变更范围

- `connection.js` 子域拆分：启动意图、消息路由、连接状态机诊断接线。
- 保持协议兼容：`{ type, payload?, meta? }` 不变。
- 保持 URL 参数兼容：`mode=create|join|ai|resume_room|resume_local|replay`。

### 上线门禁

- `npm test` 全绿。
- `depcruise` 为 error 级，无边界违规。
- `test:battle-launch-intent` 与 `test:connection-fsm` 通过。

### 回滚条件

- 任意启动模式失败率 > 1%（灰度 30 分钟窗口）。
- `welcome` 超时率显著高于基线（> 2 倍）。

### 回滚动作

1. 回滚到上一稳定 commit（仅回滚前端连接链路）。
2. 保留服务端不变，优先恢复启动可用性。
3. 复盘 `machineLastEvent` 与日志事件定位分支回归点。

## 阶段 2：性能提速（AI/开房快路径）

### 变更范围

- battle 单入口 bundle 首包路径。
- AI 模式 `welcome` 后立即发起入房，鉴权并行。
- 非 AI 路径保持保守策略，避免联机/恢复链路受影响。

### 指标阈值（建议）

- `bootstrap -> room_state`：
  - p50 <= 2.5s
  - p95 <= 5.0s
- `socket_connect_requested_at -> welcome_received_at` p95 <= 1.5s
- `launch_intent_dispatched_at -> room_state_received_at` p95 <= 2.0s

### 回滚条件

- 性能无提升且错误率上升（5xx、超时、重连失败）。
- AI 入房成功率下降超过 0.5%。

### 回滚动作

1. 切回保守入房策略（鉴权先行）。
2. 保留时序埋点，继续采样定位慢点。

## 阶段 3：稳定性与运维收口

### 变更范围

- 故障分层统一：连接/协议/房间/鉴权/回放。
- Runbook、排障模板、指标周报模板落地。
- 统一日志字段：`requestId/sessionId/roomCode/playerId`。

### 上线门禁

- CI 通过：静态检查、边界检查、协议契约、启动链集成、回放迁移。
- `healthz/readyz/metrics` 与排障文档一致。

### 回滚条件

- `readyz` 不稳定（连续 3 次失败）。
- `room_ack` 超时率明显升高（超过基线 2 倍）。

### 回滚动作

1. 先回滚服务端处理器改动。
2. 如仍异常，再回滚前端连接壳改造。
3. 用统一错误码与诊断事件复盘，避免重复故障。

## 灰度发布顺序建议

1. 内部环境（100%）
2. 小流量灰度（10%）
3. 中流量灰度（30%）
4. 全量（100%）

每个阶段至少观察 30 分钟，再推进下一档。
