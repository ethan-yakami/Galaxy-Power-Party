# 与 genius-invokation-main 的差距与待补清单

## 总判断

当前项目已经不是“随便写写的小脚本”，核心战斗引擎、服务端权威模型、回放和协议测试都已经有了明显雏形。  
但如果把本项目和 `genius-invokation-main` 放在同一个标准下比较，差距主要不在某一个 bug、某一个角色、某一个页面，而在于整体工程成熟度。

更直接地说：

- 你的项目已经能运行、能迭代、也有一定测试保护。
- 但对方项目已经明显进入“平台化工程产品”的阶段。
- 你的核心短板不是玩法逻辑，而是外围工程壳层还不够厚。
- 这些壳层包括：模块边界、类型系统、协议契约、测试深度、CI/CD、前端组件化、文档系统、对外扩展能力。

## 你明显落后的维度

### 1. 架构与模块边界

你现在已经把代码拆成了 `src/core`、`src/server`、`src/client`，方向是对的。  
但这些边界仍然偏“人为约定”，还没有被包边界、构建系统、导出规则、类型系统彻底固定住。

表现包括：

- 仍然保留 compat shim 和历史兼容路径。
- 部分运行时逻辑、协议投影、房间状态、客户端展示状态之间存在适配耦合。
- 代码分层虽然有，但很多地方仍然属于“能通就连起来”而不是“结构上不允许乱连”。

对方更强的地方：

- 明确采用 monorepo。
- `core`、`server`、`test`、`typings`、`web-client`、`web-ui-core`、`bindings`、`assets` 都是独立包。
- 能力边界不仅写在脑子里，也写进目录结构、workspace、包导出和构建链路里。

这意味着你需要补的不是“再拆几个文件”，而是补**强约束边界**。

### 2. 类型系统与接口契约

你现在的主要实现仍然是 JS，很多结构正确性依赖运行时约定和测试覆盖。  
这在项目小的时候还能维持，一旦协议、回放、客户端状态、服务端状态同时演进，风险就会迅速变高。

表现包括：

- 协议消息主要靠手写 JSON envelope。
- 很多结构是“多个地方各自知道长什么样”，不是统一单一定义来源。
- 修改 replay、message payload、room projection 时，更容易出现静默失配。

对方更强的地方：

- TypeScript 包导出明确。
- 有独立 `typings`。
- 有 `proto/` 协议定义。
- 对外接口不仅是代码实现的一部分，也是正式契约的一部分。

这类差距的后果非常现实：

- 未来迭代更容易改坏。
- 回放、客户端、服务端更容易不同步。
- 第三方工具、外部接入、多人协作的成本会明显偏高。

### 3. 测试体系深度

这部分需要公平评价：你不是没测试，甚至已经比很多同类个人项目认真得多。

当前已有的测试面包括：

- battle-engine
- protocol
- connection state machine
- replay history
- battle view model
- AI runtime

但你和对方的差距在于：  
你的测试更像“当前项目保底”，而对方已经接近“生态级测试体系”。

你当前不足的点：

- 角色/机制测试矩阵还不够厚。
- 协议测试更多是项目内回归，不是正式 schema 契约测试。
- 缺少类型层测试、跨语言层测试、工具层测试。
- 缺少更系统的 bug 回归沉淀。

对方更强的地方：

- 有独立 `packages/test`。
- 核心包自己也有测试。
- 有类型测试。
- 有 C/Python/C# binding 测试。

所以准确结论不是“你没测试”，而是：  
**你已经有基础测试能力，但测试护城河明显不如对方厚。**

### 4. 工程化与交付能力

这是你最明显的弱项之一。

你现在的项目状态更像：

- 本地能跑
- 本地能测
- 可以手动打包
- 可以继续开发

但它还不像一个“持续交付的软件产品”。

当前缺口包括：

- 没有 GitHub Actions。
- 没有自动测试流水线。
- 没有自动化构建校验。
- 没有安全扫描。
- 没有统一发布链路。
- 没有跨平台构建流程。

对方更强的地方：

- CI 流水线完整。
- 有 CodeQL。
- 有构建、测试、部署、发布。
- 有 npm / PyPI / 多平台 binding 产物路线。

这类差距带来的问题是：

- 更依赖你本人手工维护。
- 更容易出现“本地没问题，集成有问题”。
- 项目难以稳定扩交给别人维护。

### 5. 前端架构与产品化成熟度

你现在的前端是典型的多 HTML + ordered scripts + vanilla JS 页面体系。  
这种做法起步快、心智成本低、改东西直观，但天花板比较明显。

当前短板包括：

- 组件复用能力弱。
- 状态组织不够标准化。
- 页面间共享逻辑容易散落。
- 测试前端行为的成本偏高。
- UI 设计语言不够系统化。
- 多端/移动端演进能力一般。

对方更强的地方：

- `web-client` 和 `web-ui-core` 明确分离。
- 页面、布局、组件、语言、本地资源都有清晰层次。
- UI 已经不只是“能显示”，而是“可以被维护、复用、演化”。

所以这不是“现代框架就一定更高级”，而是：  
**对方把前端当成了长期产品层，你现在更多还是围绕功能搭页面。**

### 6. 文档、生态与二次开发能力

你现在不是没有文档，但文档更像“内部维护说明”和“阶段性分析”。  
这对你自己继续开发有帮助，但不足以支撑别人系统接手。

当前不足：

- 缺少系统开发文档总览。
- 缺少协议文档。
- 缺少 replay 格式说明。
- 缺少扩展角色/骰子/机制的正式说明。
- 缺少面向外部开发者的接入指南。

对方更强的地方：

- 有 development docs。
- 有 data docs。
- 有 process/state/query/log 等专题文档。
- 有多语言 README。
- 有绑定与外部使用说明。

这会直接影响：

- 别人理解你的项目有多快。
- 别人敢不敢改。
- 别人能不能扩展。

## 你并不差的地方

为了避免结论失真，这里也要把你已经做对的部分记下来。

### 1. 核心战斗逻辑已经开始纯化

`src/core/battle-engine` 的方向是对的。  
这说明你不是单纯把所有规则塞进页面脚本或房间对象里，而是在认真做纯引擎层。

### 2. 服务器权威模型是正确方向

你明确采用 server-authoritative。  
这对多人战斗、一致性、回放、AI 介入、非法动作控制都非常重要。

### 3. 已有测试链已经超过很多个人项目

你现在的测试数量虽然远不如对方，但在独立项目里不算差。  
尤其 battle-engine、protocol、AI runtime 这类测试，说明你已经开始按“可维护”方式思考，而不只是“功能先通”。

## 全部需要补的地方

下面这一部分是最重要的内容。  
它不是评价，而是后续整改清单。

## P1：必须优先补的地方

### 1. 补强模块边界

要补的内容：

- 让 `src/core` 成为真正纯规则/纯引擎层。
- 让 `src/server` 只负责连接、房间、协议、AI 调度、回放管理。
- 让 `src/client` 只负责展示、交互、前端状态。
- 逐步移除 compat shim、历史重导出和过渡性路径。
- 建立 import 方向规则，避免层间反向依赖。

目标：

- 以后新增机制时，不需要顺手改到三层以上。
- 以后排 bug 时，可以明确知道问题落在哪一层。

### 2. 补 TypeScript 与静态类型系统

要补的内容：

- 优先把协议层迁成 TypeScript。
- 给 replay schema、battle action、room state、client state 定义类型。
- 把 shared schema 统一成单一定义来源。
- 为 public API 和核心数据结构建立类型导出。

目标：

- 让改结构时尽量先在编译阶段报错。
- 降低“看起来没问题，运行时才炸”的概率。

### 3. 补正式协议契约层

要补的内容：

- 明确所有 WebSocket 消息类型。
- 为每种消息定义 payload 结构、错误码、actor 约束、phase 约束。
- 给 replay 导出和恢复结构建立统一版本化定义。
- 建立集中协议文件，而不是分散在消息处理器里各写各的。

目标：

- 客户端、服务端、回放工具、外部工具共享同一协议真相。

### 4. 补更深的测试体系

要补的内容：

- 为角色和光环骰建立更系统的机制矩阵测试。
- 为 battle-engine 补随机/属性测试。
- 为协议层补 schema 测试和非法输入测试。
- 为 replay 做版本兼容测试。
- 为断线重连、房间生命周期、边界状态补集成测试。
- 为工坊/自定义内容补测试。
- 将历史 bug 全部沉淀成回归用例。

目标：

- 让未来迭代不是靠“改完手动试试看”。

### 5. 补 CI/CD

要补的内容：

- GitHub Actions 基础工作流。
- Push / PR 自动测试。
- Node 版本固定。
- 构建检查。
- 基础静态检查。
- 自动生成 portable build 或测试产物。

目标：

- 让仓库从“本地工程”变成“可持续交付工程”。

- 建立统一的 view-model 或 UI state 层。

目标：

- 让前端可以更稳定迭代，而不是越改越缠。

### 7. 补前端设计系统

- 统一资源与图标命名方式。

目标：

- 让 UI 不再只是“功能拼接”，而是具有系统性。

### 8. 补系统文档

要补的内容：

- 架构总览文档。
- battle-engine 状态流转文档。
- 消息协议文档。
- replay 文档。
- 自定义内容格式文档。
- 开发入门文档。
- 测试说明文档。

### 9. 补外部扩展说明

### 10. 补错误处理与可观测性

要补的内容：

- 完整错误码表。
- 服务端日志级别。
- AI 行为诊断日志。

- 客户端错误提示分层。

目标：

- 更容易定位线上和复杂逻辑 bug。

## P4：你不一定要立刻补的地方

### 16. 不一定现在就拆 monorepo

虽然你这方面不如对方，但这不是最急的。  
如果当前仓库边界还没清干净，过早拆 monorepo 只会把复杂度提前。

更合理顺序：

1. 先清边界
2. 再补类型
3. 再补协议
4. 再考虑是否包化

### 17. 不一定现在就做跨语言 binding

对方的 C / Python / C# binding 很强，但这是平台化后期能力。  
在你自己的项目还没把协议、类型、测试、CI 补稳前，不应该优先投入。

### 18. 不一定现在就全量迁现代前端框架

你前端确实不如对方成熟，但不代表必须立刻整体迁 React/Vue/Solid。  
更合适的顺序是：

1. 先组件化
2. 先规范状态层
3. 先统一 UI
4. 再决定要不要框架化
