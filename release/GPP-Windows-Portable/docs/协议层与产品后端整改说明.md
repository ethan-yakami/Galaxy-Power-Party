# 协议层与产品后端整改说明

## 1. 这轮整改做了什么

这轮整改没有把仓库拆成 monorepo，也没有把现有 WebSocket 运行时直接改成 protobuf 传输。
这轮的目标是把当前单仓项目补成“有协议源头、有契约测试、有产品后端骨架、有正式调试工具”的半成品工程外壳。

已经落地的核心变化：

- `proto/` 现在是协议源头，不再只有一个总 proto 文件。
- 运行时仍然兼容当前 JSON envelope：`{ type, payload?, meta? }`。
- 协议生成改成读取 `proto/*.proto`，并生成 `protocol-manifest.json` 与 `protocol-types.d.ts`。
- 新增了产品后端骨架：账号、会话、回放记录、审计事件、health/ready/metrics/debug 接口。
- 新增了正式 CLI：`protocol:inspect`、`replay:inspect`、`debug:room`、`data:check`。
- 新增了协议契约测试、服务测试、工具测试。

## 2. 协议层现状

### 2.1 source of truth

协议源头目录：

- `proto/envelope.proto`
- `proto/errors.proto`
- `proto/room.proto`
- `proto/battle.proto`
- `proto/replay.proto`
- `proto/auth.proto`

生成产物目录：

- `src/core/shared/generated/protocol-manifest.json`
- `src/core/shared/generated/protocol-types.d.ts`

运行时适配层：

- `src/core/shared/protocol/schema.js`
- `src/server/transport/protocol/messages.js`

### 2.2 这轮已经结构化的高价值字段

当前已经从字符串包装中解开的重点字段：

- `battle_actions.actions`
- `create_resume_room.replay`
- `replay_export.content`
- 错误响应和握手里的 `meta`

这意味着运行时仍然发 JSON，但这些关键 payload 不再强制走 `*_json` 字符串包裹。

### 2.3 协议检查链路

相关命令：

```bash
npm run protocol:generate
npm run protocol:check
npm run protocol:inspect -- --type welcome
```

说明：

- `protocol:generate` 会从 `proto/` 重新生成 manifest 和类型声明。
- `protocol:check` 会检查生成产物是否过期。
- 如果机器上装了 `buf`，外部 lint 会一起执行；没装时会跳过，但本地解析和校验仍然执行。

## 3. 产品后端骨架

### 3.1 新增平台层

新增目录：

- `src/server/platform/config.js`
- `src/server/platform/create-platform.js`
- `src/server/platform/http.js`
- `src/server/platform/metrics.js`
- `src/server/platform/passwords.js`
- `src/server/platform/tokens.js`
- `src/server/platform/store/*`
- `prisma/schema.prisma`

当前策略：

- 默认使用内存 store，保证本地开发直接可跑。
- 如果后续补上 Prisma 依赖并配置 `DATABASE_URL`，可以切到 `prisma` provider。
- 直播房间和 battle state 仍是内存态，这轮没有做房间持久化。

### 3.2 认证与会话

当前接口：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`
- `GET /api/me`

当前模型：

- 本地用户名密码注册/登录
- `access token + refresh token`
- access token 既可用于 HTTP，也可通过 WebSocket URL query 透传给服务端
- 匿名用户仍然允许本地/AI 对战
- 登录用户的回放导出会进入 replay record 存储

### 3.3 运维与诊断接口

当前接口：

- `GET /api/healthz`
- `GET /api/readyz`
- `GET /api/metrics`
- `GET /api/debug/rooms`
- `GET /api/debug/rooms/:roomCode`
- `GET /api/replays`

说明：

- `healthz` 用于活性检查。
- `readyz` 用于依赖就绪检查。
- `metrics` 输出 Prometheus 文本格式。
- `debug/rooms` 提供房间、玩家、连接的机器可读诊断快照。
- `replays` 返回当前登录用户的 replay 记录摘要。

## 4. 开发调试工具

### 4.1 正式 CLI

```bash
npm run protocol:inspect -- --type welcome
npm run replay:inspect -- --file tmp/sample-replay.json
npm run debug:room -- --base-url http://127.0.0.1:3000 --room 1234

```

对应脚本：

- `tools/dev/protocol_inspect.js`
- `tools/dev/replay_inspect.js`
- `tools/dev/debug_room.js`
- `tools/dev/data_check.js`

### 4.2 CLI 输出约定

这些 CLI 都按统一思路输出：

- 标准输出为 machine-readable JSON
- 失败时返回稳定 exit code
- 脚本既可以直接命令行执行，也可以被测试直接调用

## 5. 回放与持久化

这轮补到的范围：

- WebSocket `export_replay` 仍然保持原有运行时语义
- `replay_export.content` 现在可以直接发对象
- 已登录用户导出 replay 时，服务端会把 replay 元信息写入平台存储
- replay 记录最少包含：
  - `ownerUserId`
  - `replayId`
  - `version`
  - `createdAt`
  - `sourceRoomMode`

## 7. 这轮还没做完的部分

这轮已经把骨架搭起来了，但还没有到“对标成熟平台后端”的终态。

仍然没做完的关键点：

- `proto/` 还没有像对方那样彻底拆到 action/state/rpc/notification 细粒度层级
- 还没有真正启用 Buf 生成链和 breaking rule 作为强制门禁
- Prisma 还是 schema + provider 骨架，默认运行仍是 memory store
- 还没有账号权限模型、后台管理、房间持久化、真实观测平台接入
- 还没有独立的第三方客户端 SDK 或跨语言 binding

## 8. 下一步建议

如果继续往“更像 genius-invokation-main”推进，下一轮优先级建议是：

1. 真正补齐 Prisma 依赖、迁移脚本和数据库初始化链路
2. 把 `proto/` 再拆细，并让生成产物完全由 Buf 驱动
3. 给 auth / replay / protocol fixture 再加一层跨版本测试矩阵
4. 把 `launcher / replay / workshop` 也逐步接到统一诊断与鉴权能力上
